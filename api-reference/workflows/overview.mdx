Workflows let you define complex, branching conversation logic as a **directed graph** of nodes. You write human-readable instructions and let the platform compile them into an executable graph, or supply the graph directly.

A workflow follows this lifecycle:

1. **Create** a workflow (a named container).
2. **Create a revision** with instructions, prompts, and configuration.
3. **Compile** the revision — the platform converts instructions into a graph.
4. **Validate** — check the graph for errors before going live.
5. **Activate** the revision so agents can use it.

Workflows are attached to **agent revisions** at the agent level, not at the workflow level.

---

## Create a Workflow

```
POST https://api-sandbox.featherhq.com/v1/workflows
```

### Request Body

<ParamField body="name" type="string" required>
  Human-readable workflow name, unique within your workspace.
</ParamField>

<ParamField body="description" type="string">
  Optional description for internal documentation.
</ParamField>

### Response

Returns a **WorkflowResponse** object.

<ResponseField name="id" type="string">
  Unique workflow ID.
</ResponseField>

<ResponseField name="name" type="string">
  Workflow name.
</ResponseField>

<ResponseField name="description" type="string">
  Workflow description.
</ResponseField>

<ResponseField name="active_revision" type="object">
  The currently active revision, or `null`.

  <Expandable title="active_revision fields">
    <ResponseField name="id" type="string">Revision ID.</ResponseField>
    <ResponseField name="version" type="integer">Monotonically increasing version number.</ResponseField>
    <ResponseField name="compilation_status" type="string">
      One of `draft`, `pending`, `succeeded`, or `failed`.
    </ResponseField>
    <ResponseField name="compiled_graph" type="object">
      The compiled graph object, or `null` if not yet compiled.
    </ResponseField>
    <ResponseField name="compilation_errors" type="array">
      List of error strings from the last compilation attempt.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="attachments" type="array">
  List of agent revision IDs that currently reference this workflow's active revision.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 creation timestamp.
</ResponseField>

<ResponseField name="updated_at" type="string">
  ISO 8601 last-update timestamp.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X POST https://api-sandbox.featherhq.com/v1/workflows \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Billing Flow",
    "description": "Guides users through billing inquiries and dispute resolution"
  }'
```

```python Python
import httpx

resp = httpx.post(
    "https://api-sandbox.featherhq.com/v1/workflows",
    headers={"x-api-key": "<key>"},
    json={"name": "Billing Flow"},
)
print(resp.json())
```
</CodeGroup>

## Delete a Workflow

```
DELETE https://api-sandbox.featherhq.com/v1/workflows/{id}
```

Permanently deletes the workflow and all its revisions.

<Warning>
  Deleting a workflow detaches it from any agent revisions that reference it. Test agents before deletion to avoid broken flows.
</Warning>

## Workflow Revisions

Revisions are versioned snapshots of a workflow's instructions and configuration. Only the **active revision** is executed by agents at runtime.

### Create a Revision

```
POST https://api-sandbox.featherhq.com/v1/workflows/{id}/revisions
```

<ParamField path="id" type="string" required>
  Workflow ID.
</ParamField>

#### Request Body

<ParamField body="name" type="string">
  Optional label for this revision (e.g. `"v2 – add refund node"`).
</ParamField>

<ParamField body="instructions" type="string">
  Natural-language instructions describing the workflow's steps and branching logic. The compiler converts this into the graph. Required when `compilation_strategy` is `auto`.
</ParamField>

<ParamField body="system_prompt" type="string">
  System-level prompt prepended to every node that invokes an LLM.
</ParamField>

<ParamField body="compilation_strategy" type="string">
  How to build the graph. One of:
  - `auto` — LLM-based compilation from `instructions`.
  - `manual` — Supply the graph directly via [PUT /graph](#replace-graph-manually).
</ParamField>

<ParamField body="trigger_type" type="string">
  When the workflow activates. One of `conversation_start`, `intent_match`, or `manual`.
</ParamField>

<ParamField body="trigger_config" type="object">
  Configuration for the selected trigger type.

  <Expandable title="trigger_config fields">
    <ParamField body="intent_labels" type="array">
      List of intent strings that trigger this workflow (for `intent_match`).
    </ParamField>
    <ParamField body="confidence_threshold" type="number">
      Minimum intent confidence score (`0`–`1`) to trigger. Default `0.7`.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="router_config_id" type="string">
  ID of a saved router configuration to use for decision nodes.
</ParamField>

<ParamField body="analyzer_model_settings" type="object">
  Model settings for the workflow's analyzer (intent classification, condition evaluation).

  <Expandable title="analyzer_model_settings fields">
    <ParamField body="model" type="string">Model identifier, e.g. `"gpt-4o-mini"`.</ParamField>
    <ParamField body="temperature" type="number">Sampling temperature.</ParamField>
  </Expandable>
</ParamField>

<ParamField body="router_model_settings" type="object">
  Model settings for routing/decision nodes within the graph.

  <Expandable title="router_model_settings fields">
    <ParamField body="model" type="string">Model identifier.</ParamField>
    <ParamField body="temperature" type="number">Sampling temperature.</ParamField>
    <ParamField body="max_tokens" type="integer">Max tokens for router responses.</ParamField>
  </Expandable>
</ParamField>

<ParamField body="policy_ids" type="array">
  List of policy IDs to enforce across all nodes in this workflow.
</ParamField>

### WorkflowRevisionResponse Fields

<ResponseField name="id" type="string">
  Revision ID.
</ResponseField>

<ResponseField name="workflow_id" type="string">
  Parent workflow ID.
</ResponseField>

<ResponseField name="version" type="integer">
  Auto-incremented version number.
</ResponseField>

<ResponseField name="name" type="string">
  Revision label.
</ResponseField>

<ResponseField name="instructions" type="string">
  Source instructions used for compilation.
</ResponseField>

<ResponseField name="compilation_status" type="string">
  Current compilation state:
  - `draft` — Not yet compiled.
  - `pending` — Compilation is in progress.
  - `succeeded` — Compiled successfully; `compiled_graph` is populated.
  - `failed` — Compilation failed; see `compilation_errors`.
</ResponseField>

<ResponseField name="compiled_graph" type="object">
  The executable graph object. Only present when `compilation_status` is `succeeded`.
</ResponseField>

<ResponseField name="compilation_errors" type="array">
  List of error message strings from the last failed compilation.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 creation timestamp.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X POST \
  https://api-sandbox.featherhq.com/v1/workflows/wf_billing/revisions \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "v1 – initial",
    "instructions": "1. Greet the user. 2. Ask whether they have a billing question or a payment issue. 3. If billing question, collect their account number and answer from knowledge base. 4. If payment issue, escalate to human agent.",
    "compilation_strategy": "auto",
    "trigger_type": "conversation_start"
  }'
```
</CodeGroup>

## Graph Endpoints

The compiled graph is stored separately and can be read or replaced independently of revision metadata.

### Get Graph

```
GET https://api-sandbox.featherhq.com/v1/workflows/revisions/{id}/graph
```

Returns the `compiled_graph` object for the revision. Useful for inspecting or exporting graphs.

### Replace Graph Manually

```
PUT https://api-sandbox.featherhq.com/v1/workflows/revisions/{id}/graph
```

Replaces the graph for a revision with a manually authored graph. Sets `compilation_strategy` to `manual` and `compilation_status` to `succeeded` (after internal validation).

<ParamField path="id" type="string" required>
  Revision ID.
</ParamField>

<ParamField body="graph" type="object" required>
  A valid workflow graph object. Use [POST …/graph/validate](/api-reference/workflows/compile#validate-a-graph) to check before saving.
</ParamField>

<Note>
  Manually supplied graphs bypass the LLM compiler but are still validated for structural integrity (no orphan nodes, valid edge references, etc.).
</Note>
