A **handoff** occurs when the AI agent determines a human operator should take over the conversation. The conversation transitions to `waiting_for_human` status. The handoff record contains a **packet** — all the context the operator needs to seamlessly continue without asking the user to repeat themselves.

---

## List Handoffs

```
GET https://api-sandbox.featherhq.com/v1/hitl/handoffs
```

Returns a cursor-paginated list of handoff requests.

<ParamField query="status" type="string">
  Filter by handoff status. One of `pending` (waiting for an operator), `active` (operator engaged), `resolved` (completed), `abandoned`.
</ParamField>

<ParamField query="cursor" type="string">
  Pagination cursor from a previous response.
</ParamField>

<ParamField query="limit" type="integer">
  Results per page. Default `20`, max `100`.
</ParamField>

### HandoffPage Fields

<ResponseField name="items" type="array">
  List of **HandoffSummary** objects.

  <Expandable title="HandoffSummary fields">
    <ResponseField name="id" type="string">Handoff ID.</ResponseField>
    <ResponseField name="conversation_id" type="string">Associated conversation ID.</ResponseField>
    <ResponseField name="status" type="string">Current handoff status.</ResponseField>
    <ResponseField name="transfer_to" type="string">Target queue or team label, if specified.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 timestamp.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="next_cursor" type="string">
  Cursor for the next page. `null` when exhausted.
</ResponseField>

<CodeGroup>
```bash cURL
curl "https://api-sandbox.featherhq.com/v1/hitl/handoffs?status=pending&limit=20" \
  -H "x-api-key: <key>"
```
</CodeGroup>

## Get a Handoff

```
GET https://api-sandbox.featherhq.com/v1/hitl/handoffs/{id}
```

Returns the full **HandoffRead** object including the context packet the operator needs.

<ParamField path="id" type="string" required>
  Handoff ID.
</ParamField>

### HandoffRead Fields

<ResponseField name="id" type="string">
  Handoff ID.
</ResponseField>

<ResponseField name="conversation_id" type="string">
  Associated conversation ID.
</ResponseField>

<ResponseField name="status" type="string">
  `pending`, `active`, `resolved`, or `abandoned`.
</ResponseField>

<ResponseField name="packet" type="object">
  The **HandoffPacket** — all context bundled by the agent to help the operator. See [HandoffPacket](#handoffpacket) below.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp when the handoff was created.
</ResponseField>

<ResponseField name="resolved_at" type="string">
  ISO 8601 timestamp when the handoff was resolved. `null` if still active.
</ResponseField>

<CodeGroup>
```bash cURL
curl https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123 \
  -H "x-api-key: <key>"
```
</CodeGroup>

## HandoffPacket

The `packet` object is populated by the agent at the moment it triggers the handoff. It gives the operator immediate context without needing to read the full transcript.

<ResponseField name="static_text" type="string">
  A pre-written message from the agent summarising why the handoff occurred (e.g. `"The customer is requesting a refund over our automated limit. Human review required."`).
</ResponseField>

<ResponseField name="reason" type="string">
  Machine-readable reason code for the handoff (e.g. `"refund_limit_exceeded"`, `"escalation_requested"`, `"policy_block"`).
</ResponseField>

<ResponseField name="context_vars" type="object">
  Snapshot of the conversation's context variables at handoff time. Includes any data collected by the agent (e.g. `account_id`, `order_number`).

  ```json
  {
    "account_id": "ACC-98765",
    "order_id": "ORD-00123",
    "refund_amount": 149.99,
    "issue_type": "duplicate_charge"
  }
  ```
</ResponseField>

<ResponseField name="summary" type="string">
  AI-generated natural-language summary of the conversation so far. Typically 2–4 sentences covering the user's issue and any steps already taken.
</ResponseField>

<ResponseField name="transcript_ptr" type="object">
  A pointer to the relevant portion of the transcript.

  <Expandable title="transcript_ptr fields">
    <ResponseField name="conversation_id" type="string">
      Conversation ID (same as handoff's `conversation_id`).
    </ResponseField>
    <ResponseField name="up_to_seq" type="integer">
      Sequence number of the last message at handoff time. Use this with [GET /messages?since_seq](/api-reference/conversations/turns#poll-for-new-messages) to retrieve the transcript.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="transfer_to" type="string">
  Optional routing hint indicating which queue or team the agent recommends (e.g. `"billing_specialists"`, `"tier_2_support"`). Your routing system can use this to assign the handoff to the right operator.
</ResponseField>

## Send a Message as Operator

```
POST https://api-sandbox.featherhq.com/v1/hitl/handoffs/{id}/messages
```

Relays a message from the human operator to the end user. The message is appended to the conversation transcript with `role: operator`.

<ParamField path="id" type="string" required>
  Handoff ID. Must have `status: active` or `pending`.
</ParamField>

### Request Body

<ParamField body="content" type="string" required>
  The operator's message text to send to the end user.
</ParamField>

### Response: HandoffMessageRead

<ResponseField name="id" type="string">
  Message ID.
</ResponseField>

<ResponseField name="handoff_id" type="string">
  Parent handoff ID.
</ResponseField>

<ResponseField name="seq" type="integer">
  Sequence number of this message within the conversation.
</ResponseField>

<ResponseField name="role" type="string">
  Always `operator` for messages sent via this endpoint.
</ResponseField>

<ResponseField name="content" type="string">
  Message text.
</ResponseField>

<ResponseField name="authored_by" type="string">
  ID of the API key or user who sent the message.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X POST \
  https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/messages \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"content": "Hi! I can see the duplicate charge on your account. I am processing the refund now — please allow 3-5 business days."}'
```

```python Python
import httpx

resp = httpx.post(
    "https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/messages",
    headers={"x-api-key": "<key>"},
    json={"content": "Hi! I can see the issue. Processing your refund now."},
)
msg = resp.json()
print(f"Message seq {msg['seq']} sent at {msg['created_at']}")
```
</CodeGroup>

## Resolve a Handoff

```
POST https://api-sandbox.featherhq.com/v1/hitl/handoffs/{id}/resolve
```

Closes the handoff and ends the conversation with a `HANDOFF_RESOLVED` status. Call this when the operator has finished helping the user.

<ParamField path="id" type="string" required>
  Handoff ID.
</ParamField>

<ParamField body="resolution_note" type="string">
  Optional internal note describing how the issue was resolved. Stored in the handoff record for audit purposes.
</ParamField>

<CodeGroup>
```bash cURL
curl -X POST \
  https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/resolve \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"resolution_note": "Refund of $149.99 processed. Case closed."}'
```
</CodeGroup>

## Full Operator Workflow (curl walkthrough)

```bash
# 1. Poll for pending handoffs
curl "https://api-sandbox.featherhq.com/v1/hitl/handoffs?status=pending" \
  -H "x-api-key: <key>"
# → Returns list with handoff IDs

# 2. Fetch the handoff packet to understand the context
curl https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123 \
  -H "x-api-key: <key>"
# → Review packet.summary, packet.context_vars, packet.transfer_to

# 3. Retrieve the transcript up to handoff point
curl "https://api-sandbox.featherhq.com/v1/v2/conversations/conv_xyz456/messages?since_seq=-1" \
  -H "x-api-key: <key>"

# 4. Send an opening message to the user
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/messages \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"content": "Hi, I am Sarah from the billing team. I can see the duplicate charge — let me fix that for you."}'

# 5. Continue the conversation (send more messages as needed)
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/messages \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"content": "Your refund of $149.99 has been submitted. You should see it in 3–5 days."}'

# 6. Resolve when done
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/handoffs/hnd_abc123/resolve \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"resolution_note": "Duplicate charge confirmed and refund issued."}'
```

<Tip>
  Poll `GET /v1/v2/conversations/{id}/messages?since_seq={last_seq}` while the handoff is active to receive new messages from the end user in real time and display them in your operator console.
</Tip>

## Failed Warm Transfers

If your agent attempts a **warm transfer** (live call transfer to an external number or queue) and the transfer fails, the failure is recorded separately.

### List Failed Transfers

```
GET https://api-sandbox.featherhq.com/v1/hitl/failed-transfers
```

Returns a paginated list of warm transfer failures.

<ParamField query="cursor" type="string">
  Pagination cursor.
</ParamField>

<ParamField query="limit" type="integer">
  Results per page. Default `20`, max `100`.
</ParamField>

### FailedTransfer Fields

<ResponseField name="id" type="string">
  Failed transfer record ID.
</ResponseField>

<ResponseField name="conversation_id" type="string">
  Conversation in which the transfer was attempted.
</ResponseField>

<ResponseField name="transfer_to" type="string">
  Destination that was attempted (phone number, queue ID, etc.).
</ResponseField>

<ResponseField name="failure_reason" type="string">
  Why the transfer failed (e.g. `"destination_busy"`, `"no_answer"`, `"invalid_number"`).
</ResponseField>

<ResponseField name="attempted_at" type="string">
  ISO 8601 timestamp of the failed attempt.
</ResponseField>

<CodeGroup>
```bash cURL
curl "https://api-sandbox.featherhq.com/v1/hitl/failed-transfers?limit=20" \
  -H "x-api-key: <key>"
```
</CodeGroup>

<Note>
  After a failed warm transfer, the conversation typically falls back to a `waiting_for_human` handoff. Monitor this endpoint in your operator dashboard to detect routing issues.
</Note>
