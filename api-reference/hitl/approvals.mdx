**Human-in-the-Loop (HITL) Approvals** allow a human reviewer to inspect and approve or deny specific agent actions before they execute. When an approval is required, the conversation enters `awaiting_approval` status and the agent pauses until a decision is made.

Approval requests are generated by:
- **Workflow approval nodes** — a node in the compiled graph explicitly requires human sign-off before proceeding.
- **Tool call gates** — a tool is configured to require approval before its inputs are sent to the external system.

---

## List Approval Requests

```
GET https://api-sandbox.featherhq.com/v1/hitl/approvals
```

Returns a cursor-paginated list of approval requests matching the filters.

<ParamField query="status" type="string">
  Filter by approval status. One of `pending`, `approved`, `denied`, `expired`, `cancelled`. Most commonly used with `status=pending`.
</ParamField>

<ParamField query="approver_role" type="string">
  Filter by the role required to decide (e.g. `"supervisor"`, `"compliance"`, `"billing_team"`).
</ParamField>

<ParamField query="cursor" type="string">
  Pagination cursor from a previous response.
</ParamField>

<ParamField query="limit" type="integer">
  Results per page. Default `20`, max `100`.
</ParamField>

### Response: CursorPage\<ApprovalRequestSummary\>

<ResponseField name="items" type="array">
  List of **ApprovalRequestSummary** objects.

  <Expandable title="ApprovalRequestSummary fields">
    <ResponseField name="id" type="string">Approval request ID.</ResponseField>
    <ResponseField name="conversation_id" type="string">The conversation that triggered this request.</ResponseField>
    <ResponseField name="subject_kind" type="string">
      What is awaiting approval: `workflow_node` or `tool_call`.
    </ResponseField>
    <ResponseField name="subject_ref" type="string">
      Node ID or tool name being approved.
    </ResponseField>
    <ResponseField name="approver_role" type="string">Required reviewer role.</ResponseField>
    <ResponseField name="status" type="string">Current approval status.</ResponseField>
    <ResponseField name="timeout_at" type="string">ISO 8601 deadline; auto-denied after this time.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 creation timestamp.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="next_cursor" type="string">
  Cursor for the next page. `null` when exhausted.
</ResponseField>

<CodeGroup>
```bash cURL
curl "https://api-sandbox.featherhq.com/v1/hitl/approvals?status=pending&limit=20" \
  -H "x-api-key: <key>"
```
</CodeGroup>

## Get an Approval Request

```
GET https://api-sandbox.featherhq.com/v1/hitl/approvals/{id}
```

Returns the full **ApprovalRequestDetail** including the action payload the reviewer needs to make an informed decision.

<ParamField path="id" type="string" required>
  Approval request ID.
</ParamField>

### ApprovalRequestDetail Fields

<ResponseField name="id" type="string">
  Approval request ID.
</ResponseField>

<ResponseField name="conversation_id" type="string">
  ID of the paused conversation.
</ResponseField>

<ResponseField name="conv_turn_id" type="string">
  ID of the turn that triggered the approval request.
</ResponseField>

<ResponseField name="node_id" type="string">
  Workflow node ID, if `subject_kind` is `workflow_node`.
</ResponseField>

<ResponseField name="subject_kind" type="string">
  What is awaiting approval:
  - `workflow_node` — A graph node requires sign-off before the workflow proceeds.
  - `tool_call` — A tool invocation requires sign-off before the call is dispatched.
</ResponseField>

<ResponseField name="subject_ref" type="string">
  Node ID or tool name being approved.
</ResponseField>

<ResponseField name="payload" type="object">
  The data the reviewer should inspect. For `tool_call`, this is the proposed tool input. For `workflow_node`, this is the context snapshot at the point of the approval node.

  <Expandable title="Example payload (tool_call)">
    ```json
    {
      "tool_name": "process_refund",
      "inputs": {
        "account_id": "ACC-98765",
        "amount": 149.99,
        "currency": "USD",
        "reason": "Duplicate charge"
      }
    }
    ```
  </Expandable>
</ResponseField>

<ResponseField name="suspend_message" type="string">
  The message displayed to the end user while the conversation is paused (e.g. `"Your request is being reviewed. We'll be in touch shortly."`).
</ResponseField>

<ResponseField name="approver_role" type="string">
  Role label required to decide this request.
</ResponseField>

<ResponseField name="status" type="string">
  One of `pending`, `approved`, `denied`, `expired`, `cancelled`.
</ResponseField>

<ResponseField name="decision" type="string">
  `approve` or `deny`. `null` while `status` is `pending`.
</ResponseField>

<ResponseField name="decided_by" type="string">
  ID of the user or service account that made the decision.
</ResponseField>

<ResponseField name="decided_at" type="string">
  ISO 8601 timestamp of the decision. `null` while pending.
</ResponseField>

<ResponseField name="timeout_at" type="string">
  ISO 8601 deadline. After this time, the request is auto-denied and the conversation is notified.
</ResponseField>

<CodeGroup>
```bash cURL
curl https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123 \
  -H "x-api-key: <key>"
```
</CodeGroup>

## Decide on an Approval

```
POST https://api-sandbox.featherhq.com/v1/hitl/approvals/{id}/decide
```

Submits an `approve` or `deny` decision. Once decided, the conversation immediately resumes (if approved) or receives a denial message and the workflow branches to its denial path (if denied).

<ParamField path="id" type="string" required>
  Approval request ID. Must have `status: pending`.
</ParamField>

### Request Body

<ParamField body="decision" type="string" required>
  The reviewer's decision. One of:
  - `approve` — Allow the action to proceed.
  - `deny` — Block the action; the workflow follows the denial branch.
</ParamField>

<ParamField body="note" type="string">
  Optional reviewer note stored with the decision. Visible in the audit trail but not sent to the end user.
</ParamField>

### Response

Returns the updated **ApprovalRequestDetail** with `status`, `decision`, `decided_by`, and `decided_at` populated.

<CodeGroup>
```bash cURL – Approve
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123/decide \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "approve",
    "note": "Verified duplicate charge in billing system. Refund is correct."
  }'
```

```bash cURL – Deny
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123/decide \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "deny",
    "note": "Could not verify duplicate charge. Escalate to billing team."
  }'
```

```python Python
import httpx

resp = httpx.post(
    "https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123/decide",
    headers={"x-api-key": "<key>"},
    json={"decision": "approve", "note": "Verified – proceed with refund."},
)
print(resp.json()["status"])  # "approved"
```
</CodeGroup>

## Full Approval Workflow (curl walkthrough)

```bash
# 1. Poll for pending approvals assigned to your role
curl "https://api-sandbox.featherhq.com/v1/hitl/approvals?status=pending&approver_role=billing_team" \
  -H "x-api-key: <key>"

# 2. Inspect the detail of a specific request
curl https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123 \
  -H "x-api-key: <key>"

# 3. Approve it (or deny with "decision": "deny")
curl -X POST https://api-sandbox.featherhq.com/v1/hitl/approvals/apr_abc123/decide \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"decision": "approve", "note": "Refund confirmed in billing portal."}'
```

<Tip>
  Build a real-time approval queue by polling `GET /hitl/approvals?status=pending` every 10–30 seconds, or use webhooks to receive push notifications when new approval requests are created.
</Tip>

## Approval Hold Messages

You can customise the message shown to end users while their conversation is paused awaiting approval.

### Get Hold Message Configuration

```
GET https://api-sandbox.featherhq.com/v1/hitl/approval-messages
```

Returns the current workspace-level and per-role hold message templates.

### Update Hold Message Configuration

```
PUT https://api-sandbox.featherhq.com/v1/hitl/approval-messages
```

<ParamField body="default_message" type="string">
  Default hold message shown to users when no role-specific message is configured.
</ParamField>

<ParamField body="role_messages" type="object">
  Map of `approver_role` → message string. These override the default for specific roles.

  ```json
  {
    "default_message": "Your request is being reviewed. We'll respond shortly.",
    "role_messages": {
      "compliance": "Your request requires compliance review. This may take up to 2 business hours.",
      "billing_team": "Our billing team is reviewing your refund request."
    }
  }
  ```
</ParamField>

<CodeGroup>
```bash cURL
curl -X PUT https://api-sandbox.featherhq.com/v1/hitl/approval-messages \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "default_message": "Your request is being reviewed and we will be with you shortly.",
    "role_messages": {
      "billing_team": "Our billing specialists are reviewing your refund request."
    }
  }'
```
</CodeGroup>

<Note>
  Hold messages support simple template variables: `{{user_name}}`, `{{approver_role}}`, and `{{timeout_minutes}}`. These are resolved at render time against the conversation's context variables.
</Note>
