## Overview

Feather automatically classifies every ingested document with a sensitivity label. This page covers how to view, override, and approve classifications, and how to configure Access Control Lists (ACLs) that restrict retrieval based on role-based clearance levels.

**Base URL:** `https://api-sandbox.featherhq.com`

---

## Document Classification

### Get Document Classification

```
GET /v1/knowledge-base/documents/{id}/classification
```

Returns the current classification state of a document.

#### Response — `ClassificationResponse`

<ResponseField name="sensitivity" type="string">
  The assigned sensitivity label: `public`, `internal`, `confidential`, or `restricted`.
</ResponseField>

<ResponseField name="sensitivity_level" type="integer">
  Numeric representation: `0` = public · `1` = internal · `2` = confidential · `3` = restricted.
</ResponseField>

<ResponseField name="classification_source" type="string">
  How the label was determined: `ai_auto`, `manual_override`, or `approved`.
</ResponseField>

<ResponseField name="classification_status" type="string">
  Current review status: `active`, `pending_review`, or `overridden`.
</ResponseField>

<ResponseField name="classification_confidence" type="number">
  AI confidence score between `0.0` and `1.0`. Only present when `classification_source` is `ai_auto`.
</ResponseField>

<ResponseField name="manual_override_floor" type="string">
  When set, the document's sensitivity can never fall below this label even after re-classification. One of `public`, `internal`, `confidential`, `restricted`.
</ResponseField>

<ResponseField name="visibility_mode" type="string">
  Access scope: `org_wide` (all members with sufficient clearance) or `audience` (restricted to specific audience tags).
</ResponseField>

<ResponseField name="audience_tags" type="string[]">
  Present when `visibility_mode` is `audience`. List of role or team tags that can access this document.
</ResponseField>

---

### Override Document Classification

```
PATCH /v1/knowledge-base/documents/{id}/classification
```

Manually override the sensitivity label or visibility settings for a document.

<ParamField body="sensitivity" type="string">
  New sensitivity label: `public`, `internal`, `confidential`, or `restricted`. Must be at or above `manual_override_floor` if one is set.
</ParamField>

<ParamField body="visibility_mode" type="string">
  Set to `org_wide` or `audience`.
</ParamField>

<ParamField body="audience_tags" type="string[]">
  Required when `visibility_mode` is `audience`. Provide at least one tag.
</ParamField>

<ParamField body="reason" type="string">
  Human-readable justification for the override. Stored in the audit log.
</ParamField>

<Warning>
  Setting `visibility_mode: audience` without providing `audience_tags` will result in a `400 Bad Request`. Always provide tags when narrowing visibility.
</Warning>

---

### Approve a Pending Classification

```
POST /v1/knowledge-base/documents/{id}/classification/approve
```

Approves a classification that is in `pending_review` status. Optionally adjust the label before approving.

<ParamField body="sensitivity" type="string">
  Override the sensitivity label as part of the approval. Leave omitted to accept the current AI-assigned label.
</ParamField>

<ParamField body="reason" type="string">
  Justification for the approval decision. Stored in the audit log.
</ParamField>

---

### List Pending Classifications

```
GET /v1/knowledge-base/knowledge-bases/{kb_id}/classification/pending
```

Returns all documents in the knowledge base that have a `classification_status` of `pending_review`.

<ResponseField name="items" type="ClassificationResponse[]">
  Array of documents awaiting review. Each item includes the full `ClassificationResponse` plus `document_id` and `document_title`.
</ResponseField>

<ResponseField name="total" type="integer">
  Total count of pending documents.
</ResponseField>

<Tip>
  Poll this endpoint or hook it up to a webhook to build a classification review queue in your admin UI.
</Tip>

---

### Re-run AI Classification

```
POST /v1/knowledge-base/knowledge-bases/{kb_id}/reclassify
```

Triggers the AI classifier to re-evaluate **all** documents in the knowledge base. Useful after updating classification rules or onboarding a new model.

<ResponseField name="job_id" type="string">
  ID of the background reclassification job.
</ResponseField>

<ResponseField name="documents_queued" type="integer">
  Number of documents queued for reclassification.
</ResponseField>

<Note>
  Reclassification runs asynchronously. Documents remain accessible at their current sensitivity label until the job completes. Documents with a `manual_override_floor` will never be reclassified below that floor.
</Note>

## ACL Configuration

The ACL system maps user roles to clearance levels, controlling which documents each role can retrieve from any knowledge base.

### Get ACL Config

```
GET /v1/knowledge-base/acl/config
```

#### Response — `ACLConfigResponse`

<ResponseField name="role_clearance_map" type="object">
  A map of `role_name → sensitivity_level`. Agents and users with the given role can only retrieve documents at or below this sensitivity level.

  ```json
  {
    "support_agent": 1,
    "team_lead": 2,
    "admin": 3
  }
  ```
</ResponseField>

<ResponseField name="classification_mode" type="string">
  How ACL enforcement is applied:

  | Value | Behaviour |
  |---|---|
  | `auto` | Feather automatically classifies and enforces ACL on every retrieval |
  | `kb_default` | Each KB has its own default clearance floor; documents without explicit labels inherit it |
</ResponseField>

---

### Update ACL Config

```
PUT /v1/knowledge-base/acl/config
```

<ParamField body="role_clearance_map" type="object" required>
  Updated role-to-clearance mapping. Provide the full map — this is a full replacement, not a merge.

  Clearance levels: `0` = public · `1` = internal · `2` = confidential · `3` = restricted.
</ParamField>

<ParamField body="classification_mode" type="string" required>
  Set to `auto` or `kb_default`.
</ParamField>

## ACL Rollout

ACL enforcement is deployed via a controlled rollout process. This prevents accidental access disruptions when enabling for the first time or after major changes.

### Preflight Check

```
POST /v1/knowledge-base/acl/rollout/preflight
```

Validates your ACL config before enabling enforcement. Returns a list of warnings (e.g. roles with no clearance mapping, documents with no label).

<ResponseField name="ready" type="boolean">
  `true` if no blocking issues were found.
</ResponseField>

<ResponseField name="warnings" type="object[]">
  Non-blocking issues that may cause unexpected behaviour.

  <Expandable title="warning fields">
    <ResponseField name="code" type="string">
      Machine-readable warning code, e.g. `role_no_clearance`, `unlabeled_documents`.
    </ResponseField>
    <ResponseField name="message" type="string">
      Human-readable description.
    </ResponseField>
    <ResponseField name="affected_count" type="integer">
      Number of roles or documents affected.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="errors" type="object[]">
  Blocking issues that must be resolved before enabling ACL. Same shape as `warnings`.
</ResponseField>

---

### Get Rollout Status

```
GET /v1/knowledge-base/acl/rollout/status
```

<ResponseField name="acl_enabled" type="boolean">
  Whether ACL enforcement is currently active for this org.
</ResponseField>

<ResponseField name="enabled_at" type="string">
  ISO 8601 timestamp when ACL was last enabled. `null` if never enabled.
</ResponseField>

<ResponseField name="last_preflight_at" type="string">
  Timestamp of the most recent preflight check.
</ResponseField>

---

### Enable ACL

```
POST /v1/knowledge-base/acl/rollout/enable
```

Activates ACL enforcement across the organisation. Run preflight first and resolve all errors.

<Warning>
  Enabling ACL immediately restricts document retrieval for all agents. Agents without a matching role in `role_clearance_map` will be unable to retrieve any documents above sensitivity level `0` (public).
</Warning>

---

### Disable ACL

```
POST /v1/knowledge-base/acl/rollout/disable
```

Disables ACL enforcement. All documents become retrievable by all agents until ACL is re-enabled.

## Examples

<CodeGroup>

```bash Get classification
curl https://api-sandbox.featherhq.com/v1/knowledge-base/documents/doc_7e4f2d1a/classification \
  -H "x-api-key: YOUR_API_KEY"
```

```bash Override classification
curl -X PATCH https://api-sandbox.featherhq.com/v1/knowledge-base/documents/doc_7e4f2d1a/classification \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "sensitivity": "confidential",
    "visibility_mode": "audience",
    "audience_tags": ["hr_team", "legal"],
    "reason": "Contains salary band information"
  }'
```

```bash Update ACL config
curl -X PUT https://api-sandbox.featherhq.com/v1/knowledge-base/acl/config \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "role_clearance_map": {
      "support_agent": 1,
      "team_lead": 2,
      "admin": 3
    },
    "classification_mode": "auto"
  }'
```

```bash Run preflight
curl -X POST https://api-sandbox.featherhq.com/v1/knowledge-base/acl/rollout/preflight \
  -H "x-api-key: YOUR_API_KEY"
```

</CodeGroup>
