---
title: "Teams: Group Agents with Supervisor or Swarm Routing"
sidebarTitle: "Teams"
description: "Create teams of agents with supervisor routing (one agent picks the best member) or swarm routing (agents hand off between each other)."
---

Teams let you group multiple agents (called **members**) under a single addressable unit. Incoming conversations are routed to the right member according to the team's **mode**:

- **`supervisor`** — A router LLM evaluates the request and picks the single best-matching member before the conversation starts.
- **`swarm`** — Agents collaborate by handing off to each other mid-conversation. The conversation migrates between members until one closes it or `swarm_max_hops` is reached.

Every change to a team's membership or routing configuration creates a **revision**. Only the active revision is used at runtime.

---

## Create a Team

<ParamField path="POST /v1/teams" type="endpoint" />

```
POST https://api-sandbox.featherhq.com/v1/teams
```

### Request Body

<ParamField body="name" type="string" required>
  Human-readable name for the team. Must be unique within your workspace.
</ParamField>

<ParamField body="description" type="string">
  Optional free-text description for internal documentation.
</ParamField>

<ParamField body="purpose" type="string">
  A natural-language statement of what this team handles. Used by the router LLM when selecting members.
</ParamField>

<ParamField body="mode" type="string" required>
  Routing mode. One of `supervisor` or `swarm`.
</ParamField>

<ParamField body="fallback_member_id" type="string">
  Assistant ID to use when the router cannot confidently select a member (supervisor mode only).
</ParamField>

<ParamField body="initial_member_id" type="string">
  Assistant ID that receives the first message before any routing occurs (swarm mode only).
</ParamField>

<ParamField body="swarm_domain_hint" type="string">
  A short phrase describing the domain (e.g. `"billing and payments"`) that helps swarm agents decide when to hand off.
</ParamField>

<ParamField body="swarm_max_hops" type="integer">
  Maximum number of agent-to-agent handoffs allowed in a single conversation (swarm mode). Defaults to `5`.
</ParamField>

<ParamField body="router_config_id" type="string">
  ID of a saved router configuration to use for member selection. Mutually exclusive with `router_model_settings`.
</ParamField>

<ParamField body="router_model_settings" type="object">
  Inline model settings for the router. Mutually exclusive with `router_config_id`.

  <Expandable title="router_model_settings fields">
    <ParamField body="model" type="string">
      Model identifier, e.g. `"gpt-4o"`.
    </ParamField>
    <ParamField body="temperature" type="number">
      Sampling temperature between `0` and `2`.
    </ParamField>
    <ParamField body="max_tokens" type="integer">
      Maximum tokens the router may generate when reasoning.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="members" type="array">
  Seed the team with an initial list of members on creation. Each element follows the [TeamMemberCreate](#team-members) schema.
</ParamField>

### Response

Returns a **TeamResponse** object.

<ResponseField name="id" type="string">
  Unique team ID.
</ResponseField>

<ResponseField name="name" type="string">
  Team name.
</ResponseField>

<ResponseField name="description" type="string">
  Team description.
</ResponseField>

<ResponseField name="mode" type="string">
  `supervisor` or `swarm`.
</ResponseField>

<ResponseField name="active_revision" type="object">
  The currently active revision, or `null` if no revision has been activated yet.

  <Expandable title="active_revision fields">
    <ResponseField name="id" type="string">Revision ID.</ResponseField>
    <ResponseField name="version" type="integer">Monotonically increasing version number.</ResponseField>
    <ResponseField name="members" type="array">List of TeamMemberResponse objects (see below).</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 timestamp.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 creation timestamp.
</ResponseField>

<ResponseField name="updated_at" type="string">
  ISO 8601 last-update timestamp.
</ResponseField>

<CodeGroup>
```bash cURL
curl -X POST https://api-sandbox.featherhq.com/v1/teams \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Support Squad",
    "mode": "supervisor",
    "purpose": "Handle billing, technical, and account queries",
    "members": [
      {
        "assistant_id": "asst_billing_01",
        "router_name": "Billing Agent",
        "router_description": "Handles invoices and payment issues",
        "position": 0,
        "enabled": true
      }
    ]
  }'
```

## Delete a Team

```
DELETE https://api-sandbox.featherhq.com/v1/teams/{id}
```

Permanently deletes the team and all its revisions. Active conversations using this team are not affected.

<Warning>
  This action is irreversible. Ensure no active conversations are routed to this team before deleting.
</Warning>

## Team Revisions

Revisions are immutable snapshots of a team's configuration. Creating a new revision does **not** automatically make it active — call [Set Active Revision](#set-active-revision) when ready.

### Create a Revision

```
POST https://api-sandbox.featherhq.com/v1/teams/{id}/revisions
```

<ParamField path="id" type="string" required>
  Team ID.
</ParamField>

The request body accepts the same routing fields as team creation: `purpose`, `mode`, `fallback_member_id`, `initial_member_id`, `swarm_domain_hint`, `swarm_max_hops`, `router_config_id`, `router_model_settings`, and an initial `members` array.

Returns a **TeamRevisionResponse**:

<ResponseField name="id" type="string">
  Revision ID.
</ResponseField>

<ResponseField name="team_id" type="string">
  Parent team ID.
</ResponseField>

<ResponseField name="version" type="integer">
  Auto-incremented version number.
</ResponseField>

<ResponseField name="members" type="array">
  Members in this revision. Empty until you add them via the members endpoints.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp.
</ResponseField>

### Get a Revision

```
GET https://api-sandbox.featherhq.com/v1/teams/revisions/{id}
```

<ParamField path="id" type="string" required>
  Revision ID.
</ParamField>

### Set Active Revision

```
POST https://api-sandbox.featherhq.com/v1/teams/{id}/set-active-revision
```

Promotes a draft revision to live. All new conversations routed to this team will use the active revision.

<ParamField path="id" type="string" required>
  Team ID.
</ParamField>

<ParamField body="revision_id" type="string" required>
  The revision to activate.
</ParamField>

<CodeGroup>
```bash cURL
curl -X POST https://api-sandbox.featherhq.com/v1/teams/team_abc123/set-active-revision \
  -H "x-api-key: <key>" \
  -H "Content-Type: application/json" \
  -d '{"revision_id": "trev_xyz789"}'
```
</CodeGroup>

## Team Members

Members are the individual agents (assistants) that belong to a team revision. Each member has routing metadata the supervisor LLM uses when picking who handles the conversation.

### List Members

```
GET https://api-sandbox.featherhq.com/v1/teams/revisions/{id}/members
```

<ParamField path="id" type="string" required>
  Revision ID.
</ParamField>

### Add a Member

```
POST https://api-sandbox.featherhq.com/v1/teams/revisions/{id}/members
```

<ParamField path="id" type="string" required>
  Revision ID.
</ParamField>

#### TeamMemberCreate fields

<ParamField body="assistant_id" type="string" required>
  ID of the assistant to add as a member.
</ParamField>

<ParamField body="assistant_name" type="string">
  Display name override for this member within the team context.
</ParamField>

<ParamField body="router_name" type="string">
  Short name the router LLM uses to identify this member (e.g. `"Billing Agent"`). Defaults to the assistant's name.

### Remove a Member

```
DELETE https://api-sandbox.featherhq.com/v1/teams/revisions/{id}/members/{member_id}
```

Removes the member from the revision. Does not affect the linked assistant.

<Note>

</Note>

## Routing Modes In Depth

### Supervisor Mode

In supervisor mode, the router LLM receives the user's opening message and the `router_name` + `router_description` for each enabled member. It selects the single best match and routes the entire conversation to that member.

<Tip>
  Providing at least 3–5 `examples` per member dramatically improves routing accuracy, especially when member domains overlap.
</Tip>

### Swarm Mode

In swarm mode, each member agent can signal a handoff by invoking a built-in transfer tool. The next agent continues the conversation from where the previous one left off. Use `swarm_max_hops` to prevent infinite loops.

The optional `swarm_domain_hint` is injected into each agent's system context so agents understand the overall team's scope before deciding to hand off.

<Warning>
  Swarm handoffs consume additional LLM calls per hop. Set `swarm_max_hops` to a reasonable value (typically `3`–`5`) to control costs.
</Warning>
