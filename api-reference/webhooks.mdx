## Overview

Feather uses [Svix](https://www.svix.com) as its webhook delivery infrastructure. Rather than managing endpoint URLs, signing secrets, and retry policies through the Feather API, you configure all of this inside the Svix App Portal — a hosted UI you access via a short-lived portal link from Feather.

**Feather provides two webhook API endpoints:**

1. `POST /v1/webhooks/portal` — Get a short-lived Svix portal URL.
2. `GET /v1/webhooks/event-catalog` — View all deliverable event types.

---

## Get the Svix Portal Link

```
POST /v1/webhooks/portal
```

Returns a short-lived URL to the Svix App Portal for your organization. Redirect your users to this URL or embed it in an `<iframe>` to let them manage their webhook endpoints without leaving your product.

### Response: PortalLinkResponse

<ResponseField name="url" type="string">
  The short-lived Svix portal URL. Valid for the duration specified in `expires_in_s`. Do not cache this URL — generate a fresh one for each session.
</ResponseField>

<ResponseField name="expires_in_s" type="integer">
  Seconds until the URL expires. Typically 3600 (1 hour).
</ResponseField>

<ResponseField name="read_only" type="boolean">
  When `true`, the portal URL opens in read-only mode — the user can view endpoints and event logs but cannot create or modify endpoints.
</ResponseField>

### Example

<CodeGroup>
```bash cURL
curl -X POST https://api-sandbox.featherhq.com/v1/webhooks/portal \
  -H "x-api-key: <key>"
```

```json Response
{
  "url": "https://app.svix.com/login#key=eyJhcHBJZCI6ImFwcF8x...",
  "expires_in_s": 3600,
  "read_only": false
}
```
</CodeGroup>

<Note>
  Endpoint URLs, signing secrets, retry policies, and event filters are all managed inside the Svix portal. The Feather API does not expose direct CRUD operations on individual endpoints.
</Note>

## Get the Event Catalog

```
GET /v1/webhooks/event-catalog
```

Returns all event types that Feather can deliver to your webhook endpoints, with descriptions, groupings, and sensitivity classifications.

### Response: EventCatalogResponse

<ResponseField name="events" type="array of objects">
  Each event entry:

  - `type` (string) — The event type identifier used in Svix filter rules.
  - `description` (string) — Human-readable description of when this event fires.
  - `group` (string) — Logical grouping: `conversation`, `knowledge_base`, `integration`, `policy`.
  - `sensitivity` (string) — Data sensitivity level: `standard` or `sensitive`. Sensitive events may contain PII and should be filtered carefully.
</ResponseField>

### Event Types

| Event Type | Group | Description |
|---|---|---|
| `conversation.session.completed` | `conversation` | A conversation session reached a terminal state (resolved, escalated, abandoned). |
| `session.extraction.completed` | `conversation` | Entity extraction on a completed session finished processing. |
| `integration.connection.error` | `integration` | An integration connection encountered an error during a tool call or token refresh. |
| `integration.connection.disconnected` | `integration` | An integration connection was intentionally or forcibly disconnected. |
| `knowledge_base.document.indexed` | `knowledge_base` | A document finished processing and is now available for retrieval. |
| `knowledge_base.document.failed` | `knowledge_base` | A document failed to process and will not be indexed. |
| `knowledge_base.document.quarantined` | `knowledge_base` | A document was quarantined due to a policy violation or unsafe content detection. |
| `policy.violation.detected` | `policy` | A policy evaluation triggered a `block` or `require_approval` action. |

### Example

<CodeGroup>
```bash cURL
curl https://api-sandbox.featherhq.com/v1/webhooks/event-catalog \
  -H "x-api-key: <key>"
```

```json Response
{
  "events": [
    {
      "type": "conversation.session.completed",
      "description": "Fired when a conversation session reaches a terminal state.",
      "group": "conversation",
      "sensitivity": "sensitive"
    },
    {
      "type": "knowledge_base.document.indexed",
      "description": "Fired when a KB document has been successfully indexed.",
      "group": "knowledge_base",
      "sensitivity": "standard"
    },
    {
      "type": "policy.violation.detected",
      "description": "Fired when a policy check triggers a block or approval action.",
      "group": "policy",
      "sensitivity": "standard"
    }
  ]
}
```
</CodeGroup>

## Using the Portal

1. Call `POST /v1/webhooks/portal` from your backend to get a fresh portal URL.
2. Redirect the user's browser to the URL, or embed it in an `<iframe>`.
3. Inside the portal, the user can:
   - Add and remove webhook endpoint URLs.
   - Configure signing secrets for HMAC verification.
   - Filter which event types each endpoint receives.
   - View delivery logs and replay failed events.

<Tip>
  Generate a new portal URL for every browser session. Portal URLs are single-use per session — sharing or bookmarking them is not secure.
</Tip>

<Warning>
  Event payloads for `conversation.session.completed` and `session.extraction.completed` may contain PII from the conversation. Ensure your webhook endpoint applies appropriate security controls.
</Warning>
