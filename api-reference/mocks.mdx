## Overview

The Mock Dispatcher lets you create HTTP mock servers that agents call in place of real external APIs. Each mock has a public URL path that Feather hosts; you configure **scenarios** to match incoming requests and return predefined responses. This is ideal for:

- Testing agent behaviour without hitting rate limits on external APIs
- Simulating error states (429, 503, timeouts)
- Demoing agents before a real integration is ready

**Base URL:** `https://api-sandbox.featherhq.com`

## Mocks

### Create a Mock

```
POST /v1/mocks
```

<ParamField body="slug" type="string" required>
  URL-safe slug for the mock, e.g. `weather-api`. This becomes part of the public URL path.
</ParamField>

<ParamField body="name" type="string" required>
  Human-readable name for this mock.
</ParamField>

<ParamField body="description" type="string">
  Optional description of what service this mock simulates.
</ParamField>

<ParamField body="default_status" type="integer" default="200">
  The HTTP status code returned when no scenario matches the request.
</ParamField>

<ParamField body="default_response_body" type="object">
  The response body returned when no scenario matches. Defaults to `{}`.
</ParamField>

<ParamField body="is_enabled" type="boolean" default="true">
  Whether the mock is active and accepting requests.
</ParamField>

#### Response — `MockResponse`

<ResponseField name="id" type="string">
  Mock identifier.
</ResponseField>

<ResponseField name="slug" type="string">
  The mock's URL slug.
</ResponseField>

<ResponseField name="name" type="string">
  Mock name.
</ResponseField>

<ResponseField name="public_url_path" type="string">
  The public URL path agents should call, e.g. `/mock/weather-api`. Prefix with the Feather sandbox base URL: `https://api-sandbox.featherhq.com/mock/weather-api`.
</ResponseField>

<ResponseField name="is_enabled" type="boolean">
  Whether the mock is accepting requests.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 creation timestamp.
</ResponseField>

### Update a Mock

```
PUT /v1/mocks/{id}
```

Full replacement of the mock's top-level fields. Does not affect scenarios.

---

### Delete a Mock

```
DELETE /v1/mocks/{id}
```

Deletes the mock, all its scenarios, and all request logs.

---

### Enable / Disable a Mock

```
POST /v1/mocks/{id}/enable
POST /v1/mocks/{id}/disable
```

Disabled mocks return `503 Service Unavailable` for all requests.

## Scenarios

Scenarios define conditional routing rules. When a request arrives at the mock, scenarios are evaluated in ascending **priority** order (1 = first). The first matching scenario wins; if none match, the mock's `default_status` and `default_response_body` are returned.

### Create a Scenario

```
POST /v1/mocks/{id}/scenarios
```

<ParamField body="name" type="string" required>
  Human-readable scenario name, e.g. `"Success – clear weather"`.
</ParamField>

<ParamField body="priority" type="integer" required>
  Evaluation order. Lower numbers are evaluated first. Gaps between priorities are allowed (e.g. 10, 20, 30) to make reordering easier.
</ParamField>

<ParamField body="match_method" type="string" default="ANY">
  HTTP method to match: `ANY`, `GET`, `POST`, `PUT`, `PATCH`, or `DELETE`.
</ParamField>

<ParamField body="match_path_pattern" type="string">
  Regex pattern matched against the request path after the mock's base path. For example, `^/forecast/.*` matches any path starting with `/forecast/`.
</ParamField>

<ParamField body="match_headers" type="object">
  Key-value pairs that must be present in the request headers. Values are matched as exact strings.
</ParamField>

<ParamField body="match_query" type="object">
  Key-value pairs that must be present as query parameters.
</ParamField>

<ParamField body="match_body" type="object">
  Key-value pairs that must be present in the request JSON body. Supports top-level key matching.
</ParamField>

<ParamField body="response_status" type="integer" required>
  HTTP status code to return when this scenario matches.
</ParamField>

<ParamField body="response_headers" type="object">
  HTTP headers to include in the response.
</ParamField>

<ParamField body="response_body" type="object">
  Response body to return as JSON.
</ParamField>

<ParamField body="delay_ms" type="integer" default="0">
  Artificial delay before responding, in milliseconds. Use to simulate slow APIs or timeouts.
</ParamField>

<ParamField body="is_enabled" type="boolean" default="true">
  Toggle this scenario on or off without deleting it.
</ParamField>

---

### List Scenarios

```
GET /v1/mocks/{id}/scenarios
```

Returns all scenarios for a mock, ordered by `priority` ascending.

---

### Replace All Scenarios (Atomic)

```
PUT /v1/mocks/{id}/scenarios
```

Atomically replaces the entire scenario list. Send an array of scenario objects (same shape as the create body). This is useful for importing scenarios from version control.

<Warning>
  This is a full replacement — any scenarios not included in the request body are deleted.
</Warning>

---

### Reorder Scenarios

```
POST /v1/mocks/{id}/scenarios/reorder
```

<ParamField body="scenario_ids" type="string[]" required>
  Array of scenario IDs in the desired priority order. The first item becomes priority 1.
</ParamField>

## Request Logs

Every request to a mock is logged. Use logs to debug why a scenario did or did not match.

### List Request Logs

```
GET /v1/mocks/{id}/requests
```

<ParamField query="cursor" type="string">
  Pagination cursor.
</ParamField>

<ParamField query="scenario_id" type="string">
  Filter to requests matched by a specific scenario.
</ParamField>

<ParamField query="since" type="string">
  ISO 8601 timestamp — only return requests after this time.
</ParamField>

<ParamField query="status" type="integer">
  Filter by response status code.
</ParamField>

<ParamField query="method" type="string">
  Filter by HTTP method.
</ParamField>

<ParamField query="path_contains" type="string">
  Filter to requests where the path contains this substring.
</ParamField>

<ParamField query="matched_only" type="boolean">
  When `true`, return only requests that matched a scenario (not the default fallback).
</ParamField>

## Example: Weather API Mock

This example creates a mock weather API with two scenarios — one for clear conditions and one for a simulated error.

### Step 1 — Create the mock

<CodeGroup>

```bash Create mock
curl -X POST https://api-sandbox.featherhq.com/v1/mocks \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "slug": "weather-api",
    "name": "Weather API Mock",
    "description": "Simulates a weather data API for agent testing.",
    "default_status": 404,
    "default_response_body": { "error": "Location not found" },
    "is_enabled": true
  }'
```

</CodeGroup>

**Response:**

```json
{
  "id": "mock_7b3a1c2d",
  "slug": "weather-api",
  "name": "Weather API Mock",
  "public_url_path": "/mock/weather-api",
  "is_enabled": true,
  "created_at": "2024-06-01T09:00:00Z"
}
```

The public URL agents call is: `https://api-sandbox.featherhq.com/mock/weather-api`

### Step 2 — Add scenarios

<CodeGroup>

```bash Sunny day scenario
curl -X POST https://api-sandbox.featherhq.com/v1/mocks/mock_7b3a1c2d/scenarios \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Success – clear weather (London)",
    "priority": 10,
    "match_method": "GET",
    "match_query": { "location": "London" },
    "response_status": 200,
    "response_body": {
      "location": "London",
      "temperature_c": 22,
      "condition": "clear",
      "humidity_pct": 45,
      "wind_kmh": 15
    },
    "delay_ms": 0,
    "is_enabled": true
  }'
```

```bash Error simulation
curl -X POST https://api-sandbox.featherhq.com/v1/mocks/mock_7b3a1c2d/scenarios \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Rate limit simulation",
    "priority": 5,
    "match_method": "ANY",
    "match_headers": { "X-Simulate-Error": "rate_limit" },
    "response_status": 429,
    "response_body": {
      "error": "Too Many Requests",
      "retry_after": 60
    },
    "delay_ms": 0,
    "is_enabled": true
  }'
```

</CodeGroup>

### Step 3 — Call the mock (as an agent would)

<CodeGroup>

```bash Call the mock
curl "https://api-sandbox.featherhq.com/mock/weather-api?location=London" \
  -H "x-api-key: YOUR_API_KEY"
```

</CodeGroup>

```json
{
  "location": "London",
  "temperature_c": 22,
  "condition": "clear",
  "humidity_pct": 45,
  "wind_kmh": 15
}
```

### Step 4 — View request logs

<CodeGroup>

```bash List logs
curl "https://api-sandbox.featherhq.com/v1/mocks/mock_7b3a1c2d/requests?matched_only=true" \
  -H "x-api-key: YOUR_API_KEY"
```

</CodeGroup>

<Tip>
  Set a high `delay_ms` (e.g. `31000`) on a scenario to simulate a timeout and test your agent's error-handling and retry behaviour without making a real slow request.
</Tip>

<Note>
  Scenarios with lower `priority` numbers are evaluated first. Put more specific matching rules (e.g. specific query params) at lower priority numbers to avoid them being shadowed by broad catch-all scenarios.
</Note>
